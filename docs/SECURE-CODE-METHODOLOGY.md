# Secure Code Methodology -- Trellis Catalog

Adapted from the [Trellis CLI Secure Code Methodology](https://github.com/NCLGISA/trellis/blob/main/docs/SECURE-CODE-METHODOLOGY.md)
for the first-party bridge catalog. This document accounts for the catalog's
role as a **content repository with executable scripts** rather than a
compiled application.

---

## 1. Principles

- **Every PR that touches bridges is validated.** CI runs manifest checks,
  Python linting, PII scanning, and structural validation on every pull
  request. No bridge change merges without passing the gate.
- **Bridge scripts are linted like production code.** The Python files in
  `tools/` are executed inside containers with API credentials. They are
  not throwaway scripts -- they deserve the same lint discipline as any
  other production code.
- **Secrets never enter the repository.** Environment files are gitignored.
  Sanitization checks block org-specific domains, IPs, hostnames, and
  credential patterns. `.env.example` files use placeholders only.
- **Dependencies are monitored per bridge.** Each bridge has its own
  `requirements.txt`. Dependency vulnerabilities are scanned and reported.
- **Content scanning replaces SAST.** The catalog's security surface is
  PII leakage and hardcoded credentials in scripts and documentation, not
  compiled-code vulnerabilities.

---

## 2. What Applies Differently from the CLI Methodology

| Area | Trellis CLI | Trellis Catalog | Rationale |
|------|-------------|-----------------|-----------|
| Language linting | golangci-lint (Go) | ruff (Python) | Bridges are Python scripts |
| Unit tests | Go `_test.go` in CI | Not in CI | Bridge tests require live API credentials |
| Vulnerability scan | govulncheck (Go stdlib) | pip-audit per bridge | Python dependency vulnerabilities |
| Release workflow | Tag-triggered cross-compile | Not applicable | Content distributed as source |
| Input validation | ValidateBridgeName | Not applicable | No user input at repo level |
| SAST | CodeQL for Go | Content scanning (PII, creds) | No compiled code to analyze |
| Dependabot | gomod, docker, actions | pip, actions | No Go or Docker deps at repo level |
| Pre-commit hooks | golangci-lint + file hygiene | ruff + file hygiene + PII patterns | Python-focused |

## 3. Python Linting

### 3.1 Requirements

All Python files under `bridges/*/tools/` SHALL pass ruff linting in CI.
Bridge scripts are production code running with elevated API credentials
inside containers.

### 3.2 Configuration

Add `ruff.toml` at the repository root:

```toml
target-version = "py311"
line-length = 120

[lint]
select = [
    "E",    # pycodestyle errors
    "W",    # pycodestyle warnings
    "F",    # pyflakes
    "I",    # isort
    "B",    # flake8-bugbear
    "S",    # flake8-bandit (security)
    "UP",   # pyupgrade
]
ignore = [
    "S101",  # assert usage (acceptable in bridge tests)
    "S603",  # subprocess without shell=True check (bridge scripts may need it)
    "S607",  # partial executable path (acceptable for system tools)
    "E501",  # line length (handled by formatter, not linter)
]

[lint.per-file-ignores]
"*_bridge_tests.py" = ["S101", "B011"]
```

### 3.3 CI Integration

Add a Python lint step to the validation workflow:

```yaml
- name: Lint Python scripts
  run: |
    pip install ruff
    ruff check bridges/*/tools/*.py
```

### 3.4 Formatting (Optional)

ruff format can enforce consistent style. Start with check-only mode
before enforcing:

```yaml
- name: Check Python formatting
  run: ruff format --check bridges/*/tools/*.py
```

## 4. CI/CD Pipeline

### 4.1 Current State

The existing `validate.yml` runs on PRs touching `bridges/**` and checks:
- bridge.yaml presence
- Required manifest fields (name, version, description, deployment.type)
- Entrypoint permissions
- SKILL.md frontmatter
- Sanitization (org-specific patterns)
- CalVer version format

### 4.2 Recommended Additions

The validation workflow should be extended with:

| Check | Priority | Type |
|-------|----------|------|
| Python linting (ruff) | High | Blocking |
| Core triad file presence | High | Blocking |
| pip-audit on requirements.txt | Medium | Advisory |
| SKILL.md required sections | Medium | Blocking |
| Dockerfile "Generated by trellis" header check | Low | Advisory |

### 4.3 Push-to-Main Validation

The current workflow only runs on PRs. Add a push trigger for main to
catch direct pushes and ensure main is always clean:

```yaml
on:
  push:
    branches: [main]
    paths:
      - "bridges/**"
  pull_request:
    paths:
      - "bridges/**"
```

### 4.4 Core Triad Validation

Every container bridge SHALL have the core triad in `tools/`:
- `{snake_name}_client.py`
- `{snake_name}_check.py`
- `{snake_name}_bridge_tests.py`

This is documented in CONTRIBUTING.md but not enforced in CI. Add:

```yaml
- name: Validate core triad
  run: |
    EXIT=0
    for bridge in ${{ steps.changed.outputs.bridges }}; do
      MANIFEST="bridges/$bridge/bridge.yaml"
      [ -f "$MANIFEST" ] || continue
      TYPE=$(grep 'type:' "$MANIFEST" | head -1 | awk '{print $2}')
      [ "$TYPE" = "host" ] && continue
      SNAKE=$(echo "$bridge" | tr '-' '_')
      for suffix in _client.py _check.py _bridge_tests.py; do
        if [ ! -f "bridges/$bridge/tools/${SNAKE}${suffix}" ]; then
          echo "ERROR: bridges/$bridge/tools/${SNAKE}${suffix} missing"
          EXIT=1
        fi
      done
    done
    exit $EXIT
```

### 4.5 Permissions

The validation workflow SHALL use minimum permissions:

```yaml
permissions:
  contents: read
```

### 4.6 No Release Workflow

The catalog does not need a release workflow. Bridges are distributed as
source via the graft mechanism. Operators build images locally with
`trellis build`. Only the trellis CLI has versioned binary releases.

## 5. Content Scanning (PII and Credentials)

### 5.1 Current State

The sanitization step checks for org-specific patterns:
- `rowancountync\.gov`
- `10\.1\.[0-9]`
- `192\.168\.[0-9]`
- `\.internal\.`
- `AKIAIOSFODNN7EXAMPLE`

### 5.2 Recommended Enhancement

Replace the hardcoded pattern list with the generic PII detection from
the Trellis CLI methodology. This makes the scanning portable across
organizations:

| Pattern | Detection | Action |
|---------|-----------|--------|
| Email domains | `@[a-zA-Z0-9]([a-zA-Z0-9-]*[a-zA-Z0-9])?\.[a-zA-Z]{2,}` with allowlist | Warn |
| RFC 1918 addresses | `\b(10\.\d{1,3}\.\d{1,3}\.\d{1,3}\|172\.(1[6-9]\|2\d\|3[01])\.\d{1,3}\.\d{1,3}\|192\.168\.\d{1,3}\.\d{1,3})\b` with allowlist | Warn |
| UUIDs | `[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}` (skip zero UUID) | Warn |
| Internal hostnames | `\b(az\d+s\d+\|srv-[a-zA-Z0-9-]+\|dc-[a-zA-Z0-9-]+)\b` | Warn |
| Hardcoded credentials | `password=`, `api_key=`, `secret=` in quoted strings | Fail |
| AWS key patterns | `AKIA[0-9A-Z]{16}` | Fail |

### 5.3 Allowlists

Allowlists prevent false positives on documentation examples:

**Email domains:** `@example.com`, `@example.org`, `@example.net`,
`@yourdomain.com`, `@contoso.com`, `@contoso.onmicrosoft.com`

**IP addresses:** `10.0.0.1`, `192.168.1.1`, `192.168.0.1`, `172.16.0.1`,
`10.0.0.0`, `192.168.0.0`, `172.16.0.0`

### 5.4 Org-Specific Patterns

In addition to the generic patterns above, each deployment of this
methodology MAY include org-specific patterns to catch leakage from the
deploying organization. These belong in a separate CI step or config file,
not hardcoded in the workflow.

## 6. Dependency Scanning

### 6.1 pip-audit

Each bridge has a `requirements.txt`. Scan all of them for known
vulnerabilities:

```yaml
- name: Scan Python dependencies
  continue-on-error: true
  run: |
    pip install pip-audit
    for req in bridges/*/requirements.txt; do
      echo "--- Scanning $req ---"
      pip-audit -r "$req" || true
    done
```

Start as advisory (`continue-on-error: true`). Promote to blocking once
all bridges are clean.

### 6.2 Dependabot

Add `.github/dependabot.yml`:

```yaml
version: 2
updates:
  - package-ecosystem: pip
    directory: /bridges/meraki
    schedule:
      interval: weekly
    commit-message:
      prefix: "deps"

  # Repeat for each bridge with a requirements.txt
  # or use a single entry if Dependabot supports glob patterns

  - package-ecosystem: github-actions
    directory: /
    schedule:
      interval: weekly
    commit-message:
      prefix: "ci"
```

Note: Dependabot's pip ecosystem requires one entry per directory.
For 21 bridges, this creates verbose configuration. An alternative is
a scheduled CI job that runs `pip-audit` across all bridges weekly.

### 6.3 Merge Policy

| Update Type | Policy |
|-------------|--------|
| GitHub Actions (minor/patch) | Merge when CI passes |
| pip dependency (patch) | Merge when CI passes |
| pip dependency (minor/major) | Review changelog, test bridge manually |

## 7. Secret Management

### 7.1 Current State (Good)

The `.gitignore` already covers:
- `.env`, `*.env.local` (credentials)
- `__pycache__/`, `*.pyc` (Python artifacts)
- `**/gsa_catalog.json` (runtime caches)

### 7.2 .env.example Files

Every bridge SHALL provide a `.env.example` with:
- All required environment variable names
- Placeholder values (never real credentials)
- Comments explaining each variable's purpose and how to obtain it

### 7.3 Credential Patterns in Scripts

The flake8-bandit rules (enabled via ruff's `S` category) will catch
common credential anti-patterns in Python code:
- Hardcoded passwords (`S105`)
- Hardcoded temp file paths (`S108`)
- Use of `exec` / `eval` (`S102`)

The sanitization CI step catches string-literal credential patterns
(`password=`, `api_key=`, `secret=`) as an additional layer.

## 8. Bridge Integration Tests

### 8.1 Nature of Bridge Tests

Bridge tests (`*_bridge_tests.py`) are **integration tests that require
live API credentials**. They cannot run in CI without secrets, and exposing
secrets in CI for 21 different platforms is impractical and risky.

### 8.2 Test Execution Model

Bridge tests are run:
- **By bridge authors** during development, inside running containers
- **By operators** after deployment, to verify connectivity and permissions
- **Not in CI** -- CI validates structure, lint, and content only

```bash
# Author/operator runs tests manually:
docker exec -it bridge-meraki \
  python3 /opt/bridge/data/tools/meraki_bridge_tests.py
```

### 8.3 What CI CAN Validate About Tests

Even without executing the tests, CI can check:
- The test file exists (core triad check)
- The test file passes Python lint (ruff)
- The test file imports and is syntactically valid (`python -m py_compile`)

```yaml
- name: Syntax-check bridge test files
  run: |
    EXIT=0
    for test_file in bridges/*/tools/*_bridge_tests.py; do
      [ -f "$test_file" ] || continue
      if ! python3 -m py_compile "$test_file" 2>&1; then
        echo "ERROR: Syntax error in $test_file"
        EXIT=1
      fi
    done
    exit $EXIT
```

## 9. Pre-commit Hooks

Add `.pre-commit-config.yaml`:

```yaml
repos:
  - repo: https://github.com/astral-sh/ruff-pre-commit
    rev: <current>
    hooks:
      - id: ruff
        args: [--fix]
        files: 'bridges/.*/tools/.*\.py$'
      - id: ruff-format
        files: 'bridges/.*/tools/.*\.py$'

  - repo: https://github.com/pre-commit/pre-commit-hooks
    rev: <current>
    hooks:
      - id: trailing-whitespace
      - id: end-of-file-fixer
      - id: check-yaml
      - id: check-merge-conflict
      - id: check-added-large-files
        args: [--maxkb=500]
```

## 10. Project Assessment Checklist

Adapted from the CLI methodology for a content/script repository.

### Python Linting
- [ ] ruff configured at repo root
- [ ] All bridge Python scripts pass lint
- [ ] Lint runs in CI as a blocking check
- [ ] flake8-bandit rules enabled for security patterns

### CI Validation
- [ ] Validation workflow runs on PR and push to main
- [ ] Manifest validation (presence, required fields, CalVer)
- [ ] Core triad enforced for container bridges
- [ ] SKILL.md frontmatter validated
- [ ] Entrypoint permissions checked
- [ ] Python syntax compilation checked
- [ ] Workflow permissions minimized (contents: read)

### Content Scanning
- [ ] Generic PII patterns (email, RFC 1918, UUID, hostnames)
- [ ] Org-specific patterns (configurable, not hardcoded)
- [ ] Hardcoded credential patterns blocked
- [ ] AWS key patterns blocked
- [ ] Allowlists configured for documentation examples

### Dependency Management
- [ ] pip-audit runs against all requirements.txt files
- [ ] Dependabot configured for GitHub Actions
- [ ] pip dependency updates tracked (Dependabot or scheduled CI)

### Secrets
- [ ] .env files gitignored
- [ ] .env.example provided for every bridge
- [ ] No hardcoded credentials in scripts (lint + CI scan)

### Bridge Tests
- [ ] Test files exist (core triad)
- [ ] Test files pass lint and syntax check
- [ ] Test execution documented in CONTRIBUTING.md
- [ ] Tests are NOT run in CI (require live credentials)
